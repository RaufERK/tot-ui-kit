Краткое описание API меню
=========================

API меню отдаёт список приложений для бокового меню.
Frontend не вычисляет права и состав меню самостоятельно: он получает от backend
готовый список приложений, доступных текущему пользователю/клиенту.

Авторизация
-----------

Отдельный Bearer token для меню не используется.

Backend определяет пользователя и клиента по своей текущей сессии/cookie.
То есть frontend просто делает GET-запрос, а backend по session cookie понимает,
какое меню надо вернуть.

Если frontend и backend находятся на разных доменах/портах, для работы
session/cookie-сценария используются cookie-запросы:

- frontend будет вызывать fetch с credentials: "include";
- backend должен вернуть Access-Control-Allow-Credentials: true;
- Access-Control-Allow-Origin должен быть конкретным origin frontend,
  например http://localhost:5173, а не "*";
- session cookie должна быть настроена так, чтобы браузер отправлял её на API.

Endpoint
--------

Основной вариант:

GET /iam/menu/

В UI-kit этот путь используется по умолчанию, если приложение передаёт
menuProps.baseUrl.

Например baseUrl = https://api.example.ru даст запрос:

GET https://api.example.ru/iam/menu/

Если в конкретном приложении нужен другой путь, frontend может передать точный
dataUrl или переопределить menuEndpoint. Для основного session/cookie-сценария
menuId не нужен.

Успешный ответ
--------------

Status: 200
Content-Type: application/json

Тело ответа - массив объектов:

[
  {
    "app_id": "dashboard",
    "app_name": "Платформа данных",
    "link": "http://localhost:5174/dashboard/",
    "description": "Главный экран",
    "available": true
  },
  {
    "app_id": "sc",
    "app_name": "Центр установок",
    "link": "https://example.ru/sc/",
    "description": "Управление системой",
    "available": true
  }
]

Поля
----

app_id - обязательное поле.
Стабильный ID приложения. По нему UI-kit выбирает локальную иконку.
Примеры: dashboard, users, sc, dwh_bridge, transformation, table_manager,
metadata, navigator, question.

app_name - обязательное поле.
Название, которое показываем в меню.

link - обязательное поле.
URL, куда перейти при клике на пункт меню. Для доступных пунктов обычно отдаётся
полный URL или рабочий относительный путь.

description - необязательное поле.
Описание приложения. Сейчас в меню не показывается, но полезно для отладки
и будущего UI.

available - необязательное поле.
Если false, frontend скроет пункт. Если поля нет, пункт считается доступным.

icon - старое поле.
UI-kit его игнорирует, потому что иконки уже зашиты в библиотеку
и выбираются по app_id.

Как frontend обрабатывает ответ
-------------------------------

- скрывает пункты с available: false;
- сохраняет порядок пунктов из массива backend;
- показывает app_name;
- при клике переходит по link;
- иконку выбирает локально по app_id;
- если app_id неизвестен, показывает fallback-иконку.

Ошибки
------

Ожидаемые статусы:

200 - меню успешно отдано. Можно вернуть пустой массив, если доступных пунктов нет.
401 - нет валидной session/cookie.
403 - сессия есть, но пользователю нельзя получать меню.
500 - ошибка backend.

Проверка реального API
----------------------

Для проверки достаточно вызвать endpoint из браузера или через curl с валидной
session cookie и убедиться, что ответ:

- имеет статус 200;
- является JSON-массивом;
- содержит поля app_id, app_name и link у каждого видимого пункта;
- не требует Authorization/Bearer token;
- возвращает только те пункты, которые доступны текущей backend-сессии;
- отдаёт пункты в нужном порядке;
- скрывает пункты с available: false.

Главное
-------

API возвращает простой JSON-массив меню для текущей сессии.
Токен для этого API не используется.
Иконки отдавать не требуется.
Права доступа и состав меню определяет backend.
